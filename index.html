Algorithms and Data Structures

For the Algorithms and Data Structures category I enhanced the Contact Service artifact from CS 320. In the original Java version contacts were stored in an ArrayList and finding one by ID required a linear search through the entire list. That approach has O(n) time complexity in the worst case, which gets slow as the number of contacts grows.

To fix that I refactored the ContactRepository to use a Python dictionary (a HashMap) so contacts are stored by contactID. This lets find_by_id run in O(1) average time instead of scanning the whole list. I also added a separate sorted_contacts list and used bisect.insort to keep it sorted by last name whenever a new contact is added. That way the get_all_contacts_sorted method (and the /contacts/sorted endpoint) can return results quickly without sorting the whole collection every single time.

These changes were important because they show real performance improvements using well-known data structures and algorithms. The HashMap makes lookups much faster for everyday use, and maintaining the sorted list with logarithmic insertion beats the cost of sorting on every request. There is a small memory trade-off for keeping both structures, but the speed gain is worth it for a service that might see many calls.
